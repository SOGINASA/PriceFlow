# MedPartners — MedArchive

**Автоматическая обработка архива прайс-листов клиник-партнёров: извлечение → нормализация → верификация → поиск цен.**

Система принимает прайсы клиник в любом формате (PDF, скан, DOCX, XLSX/XLS, CSV, ZIP-архивы), извлекает из них структуру «клиника · услуга · цена (резидент/нерезидент)», приводит названия услуг к единому справочнику, проверяет и версионирует цены и даёт REST API + веб-интерфейс для поиска «кто оказывает услугу и по какой цене».

> Хакатон 2025 · Кейс 2 «MedArchive» (Med Partners). **Сделано командой ITshechka.**

## ✨ Возможности

**Извлечение данных (любой формат)**
- PDF (текстовый) — `pdfplumber` / `PyMuPDF`, таблицы и текст;
- PDF-сканы и фото — OCR через **EasyOCR** (рус/каз кириллица + англ), авто-fallback с текстового PDF на OCR;
- DOCX (с принятием правок), XLSX/XLS, CSV;
- **ZIP-архивы** — распаковываются и обрабатываются пофайлово;
- оригиналы файлов сохраняются (для аудита и повторной обработки).

**Нормализация к справочнику + дообучение**
- точное совпадение → синонимы → нечёткий поиск (`rapidfuzz`) → семантика (эмбеддинги `sentence-transformers`);
- порог автосопоставления настраивается; что ниже порога — уходит в очередь ручной разметки;
- **дообучение**: правки оператора пополняют словарь синонимов (`LearnedSynonym`) и улучшают будущее сопоставление;
- LLM-консолидация справочника (опционально, через Groq).

**Валидация, верификация и версионирование цен**
- авто-проверки: цена > 0, нерезидент ≥ резидент, непустое название, дата не в будущем, дедупликация, аномалия при скачке цены > 50 %, конвертация валют в KZT по курсу на дату;
- очередь верификации для оператора: ручное сопоставление, подтверждение/отклонение/**исправление** цены аномалий;
- история цен по датам — хранится бессрочно и **видна всем пользователям**.

**Поиск и витрина**
- поиск **услуги → список клиник с ценами** (резидент/нерезидент, сортировка по цене);
- поиск клиник, карточка клиники с полным прайсом, контактами и поиском по позициям;
- полнотекстовый поиск (регистронезависимый, в т.ч. кириллица), многословные запросы.

**Кабинет партнёра (клиники сами ведут прайс)**
- добавление/редактирование услуг и цен, история изменений по датам;
- редактирование профиля клиники; правки сразу видны всем.

**Аналитика и дашборд**
- метрики: документы по статусам, % нормализации, **% верификации**, очереди;
- **журнал ошибок и проверок** (с логами парсинга по документам);
- панель **дообучения** (выучено синонимов, распределение по методам сопоставления);
- сравнение цен по услуге, разрезы по городам, тренды, отчёты по обработкам.

**Веб-приложение**
- React-интерфейс: лендинг (RU/KZ/EN), вход (email/пароль + демо-биометрия), рабочая область с сайдбаром/топбаром;
- адаптив + **PWA** (устанавливается как приложение, нижняя навигация на мобиле);
- OpenAPI/Swagger документация API.

## 🚀 Быстрый старт

Нужно поднять **бэкенд** (порт `5252`) и **фронтенд** (порт `3000`, проксирует `/api` на бэкенд).

### Вариант A — Docker (рекомендуется)

Поднимает PostgreSQL + Redis + backend + Celery-worker:

```bash
cd backend
cp .env.example .env            # при желании поменяйте секреты/пароли
docker compose up --build
```

API будет на `http://localhost:5252/api`, Swagger — `http://localhost:5252/api/docs`.
OCR/семантика работают «из коробки» (Tesseract не нужен — OCR через EasyOCR, PDF рендерится PyMuPDF).

### Вариант B — локально без Docker

**Бэкенд** (Python 3.11+):

```bash
cd backend
python -m venv .venv
source .venv/bin/activate        # Windows: .venv\Scripts\activate
pip install -r requirements.txt

cp .env.example .env
# Быстрый старт: закомментируйте DATABASE_URL в .env → будет SQLite (файл database/dev.db)

python seed_data.py --reset --demo   # чистая БД + демо-данные (клиники, услуги, цены, учётки)
python app.py                        # http://localhost:5252/api
```

> **PostgreSQL локально без прав администратора (Windows):** в комплекте portable-скрипты —
> `./start_db.ps1` (поднять на localhost:5432, данные в `.localdb/`), `./stop_db.ps1`, `./uninstall_postgres.ps1`.
> Тогда оставьте `DATABASE_URL=postgresql://medarchive:medarchive@localhost:5432/medarchive` в `.env`.

> **Без Redis** обработку загрузки запускайте синхронно — фронтенд так и делает (`POST /api/archives?sync=1`).
> С Redis запустите воркер: `celery -A services.tasks.celery_app worker --loglevel=info`.

### Запуск фронтенда

```bash
cd frontend
npm install
npm start                        # http://localhost:3000 (проксирует /api → :5252)
# продакшн-сборка:
npm run build
```

Открыть `http://localhost:3000`. Вход — по демо-доступам ниже или через демо-OAuth/биометрию на странице входа.

## 🔄 Конвейер обработки документа

`pending → processing → done | needs_review | error`

1. **Приём.** ZIP распаковывается; на каждый файл создаётся `PriceDocument`, оригинал сохраняется.
2. **Извлечение.** По формату выбирается экстрактор; пустой текст в PDF → авто-OCR (`scan_pdf`).
3. **Валидация строки.** Пустое имя — пропуск; цена ≤ 0 — на ревью; нерезидент < резидента — предупреждение.
4. **Конвертация валют** в KZT по курсу на дату прайса (оригинал сохраняется).
5. **Дедуп/версионирование.** Прежняя активная цена этой услуги архивируется в `PriceItemHistory`.
6. **Аномалии.** Скачок цены > 50 % → флаг, позиция в очередь ревью.
7. **Нормализация.** Сопоставление со справочником; ≥ порога — авто, иначе очередь `unmatched`.
8. **Итог.** Всё нормализовано без предупреждений → `done`, иначе `needs_review`; нет данных → `error`.

## 👤 Роли пользователей

- **Пользователь** — поиск услуг/клиник, цены, карточки клиник.
- **Партнёр (клиника)** — кабинет: свой прайс (добавление/редактирование цен с историей), профиль клиники.
- **Оператор / Администратор** — загрузка архивов, очередь верификации, дашборд и аналитика, дообучение.

JWT выдаётся `POST /api/admin/login`, передаётся заголовком `Authorization: Bearer <token>` (фронт делает автоматически). Очереди верификации защищены ролью оператора/админа.

| `GET`/`POST` | `/api/rates` · `/rates/refresh` · `/rates/convert` | курсы валют (вкл. обновление НБ РК) |
| `POST` | `/api/admin/login` | вход (JWT) |

## 🗃 Модель данных

| Сущность | Назначение |
|----------|-----------|
| **Service** | целевой справочник услуг (название, синонимы, категория, МКБ) |
| **Partner** | клиника-партнёр (название, город, адрес, БИН, контакты, описание) |
| **PriceDocument** | исходный прайс-документ (файл, формат, дата, статус парсинга, лог, сырой текст) |
| **PriceItem** | позиция прайса (услуга-сырьё + ссылка на Service, цены резидент/нерезидент, валюта, score, флаги) |
| **PriceItemHistory** | архив версий цены (бессрочно — история по датам) |
| **User** | учётная запись (роль; для партнёра — привязка к клинике) |
| **LearnedSynonym** | синонимы, выученные на правках оператора (дообучение) |
| **ExchangeRate** | курсы валют по датам |

## 🛠 Траблшутинг

- **`database is locked` (SQLite).** Возникает при двух одновременных писателях. Не запускайте два инстанса бэка на одной `dev.db`; не гоняйте параллельно несколько тяжёлых загрузок. Для нагрузки/прод используйте PostgreSQL (`DATABASE_URL`). Застрявшие документы чинит `python recover_stuck.py`.
- **Первая обработка медленная.** EasyOCR (~80 МБ/язык) и семантическая модель (~120 МБ) скачиваются при первом запуске, дальше работают офлайн. На слабом железе/для демо: `SEMANTIC_MATCH_ENABLED=false`.
- **Большие сканы.** OCR PDF-сканов идёт минуты — это нормально; на экране «Анализ» прогресс «упирается» в ~92 %, пока бэкенд реально распознаёт, затем доходит до 100 %.
- **Фронт не видит бэк.** Проверьте, что бэкенд на `:5252` и в `frontend/package.json` стоит `"proxy": "http://localhost:5252"` (для dev) либо задан `REACT_APP_API_URL`.
